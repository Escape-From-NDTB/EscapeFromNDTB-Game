# Контекст проекта: Escape From NDTB

> Этот документ задаёт архитектурные правила для всего кода игры.  
> При добавлении новых систем или правке существующих строго следуй описанным ниже соглашениям.

## 1. Общие принципы

- **Язык:** C# (Для разработки игры используется Unity 6.4, а именно 6000.4.10f1).
- **Парадигма:** компонентно-ориентированная с элементами событийно-ориентированного подхода.  
- **Связи между системами:** через `EventBus` (слабое связывание). Прямые ссылки допускаются только внутри одной системы (например, `PlayerController` ссылается на `PlayerStats`).  
- **Данные:** все настройки, баланс, диалоги и рецепты хранятся в `ScriptableObject`. Скрипты работают с экземплярами SO, никогда не хардкодят значения.  
- **Сцены:** аддитивная загрузка. Корневая сцена `SCN_Persistent` всегда загружена, в ней находятся менеджеры. Локации и UI подгружаются поверх.  
- **Сохранение:** через `JsonUtility` в `Application.persistentDataPath`. Сохраняются только примитивные типы и сериализуемые классы (без MonoBehaviour).  
- **Нейминг:**  
  - Префабы: `PRF_Имя`  
  - Скрипты: `ИмяСистемы` (без суффиксов)  
  - ScriptableObject: `SO_Имя`  
  - Сцены: `SCN_Имя`  
  - События: `OnИмяСобытия` (для EventBus)  
- **Пространства имён:** использовать вложенные по папкам, например `NDTB.Core`, `NDTB.Systems.Player`.

## 3. Ключевые системы и их обязанности

### 3.1. GameManager (Core)
- Синглтон (через `DontDestroyOnLoad`).
- Содержит ссылки на все менеджеры (по возможности через интерфейсы).
- Управляет состоянием игры (пауза, загрузка сохранения).
- Отвечает за инициализацию систем при старте.

### 3.2. EventBus (Core)
- Статический класс или синглтон для глобальных событий.
- Подписка: `EventBus.Subscribe<OnPlayerCaught>(SomeMethod)`.
- Вызов: `EventBus.Publish(new OnPlayerCaught())`.
- Используется для оповещения о: смена времени суток, получение объяснительной, изменение настроения, активация лифта, завершение крафта, смена локации и т.д.

### 3.3. Система игрока (Systems/Player)
- **PlayerController**: управление движением (Rigidbody + Cinemachine), взаимодействие с объектами (триггеры, клики).
- **PlayerStats**: хранит текущее настроение (0–100), количество объяснительных, максимальные слоты инвентаря. Методы `ChangeMood(int delta)`, `AddExplanation()`, `CheckGameOver()`.
- **InventoryManager**: список предметов (ограниченное количество), методы `AddItem(ItemSO item)`, `RemoveItem(string id)`, `HasItem(string id)`.
- **JournalSystem**: хранение заметок с временем. Методы `AddNote(string text)`, `GetAllNotes()`.

### 3.4. Время и расписание (Systems/DayNight)
- **TimeManager**: управляет игровым временем (день/ночь). Переключение между "день" и "ночь" через событие `OnNightStarted`/`OnDayStarted`. Содержит метод `SkipTime()` (сон).
- **NPC_Schedule**: прикрепляется к каждому NPC-патрульному. Подписывается на события TimeManager и меняет своё поведение (маршрут, активность). Проверяет, находится ли игрок вне комнаты в ночное время, и при обнаружении вызывает объяснительную.

### 3.5. Стелс и детекция (Systems/Stealth)
- **DetectionManager**: навешивается на камеры наблюдения. При попадании игрока в триггер (или на основе LineOfSight) вызывает событие `OnPlayerDetected` с типом нарушения (разбор мебели, хождение ночью, попытка пройти турникет).  
- **NoiseEmitter**: объекты, создающие шум (разрушение, бег). Влияют на радиус обнаружения патрульных.

### 3.6. Крафт (Systems/Crafting)
- **CraftingManager**: проверяет рецепт (список нужных предметов и их количество). Вызывает `OnCraftCompleted()` при успехе. Содержит список всех доступных рецептов (SO_Recipe).  
- **Workbench**: MonoBehaviour, который игрок активирует взаимодействием. При активации открывает UI-окно крафта, которое связано с CraftingManager.

### 3.7. Диалоги (Systems/Dialogue)
- Используется Yarn Spinner (или кастомная система на SO).  
- **DialogueManager**: загружает узлы диалога, отображает их через UI. При выборе варианта может вызывать события (например, `OnDialogueOptionSelected`), которые влияют на флаги сюжета.

### 3.8. Разрушение мебели (Systems/Destruction)
- **BreakableFurniture**: содержит список частей, Rigidbody-компоненты и Fixed Joints. При активации (вызов `Break()`) разъединяет джойнты, включает Rigidbody и генерирует обломки через ObjectPool.
- **DebrisPool**: ObjectPool для обломков, чтобы не создавать мусор в памяти.

### 3.9. Катсцены (Systems/Cutscene)
- **CutsceneTrigger**: триггер, который при входе игрока запускает Timeline (с привязкой к камере Cinemachine). После окончания Timeline возвращает управление игроку. Использует `PlayableDirector`.

### 3.10. UI-система
- **UIManager**: отвечает за открытие/закрытие панелей (HUD, инвентарь, дневник, диалог).  
- **HUDController**: отображает текущее настроение, количество объяснительных, слоты инвентаря (иконки). Подписывается на события изменения стат.
- **InventoryUI**: полноэкранное окно, показывает все предметы с описанием.  
- **JournalUI**: показывает список заметок, автоматически добавляет текущее игровое время.

### 3.11. Сохранение/загрузка (Core/SaveSystem)
- Сериализует данные в JSON (состояние игрока, инвентарь, заметки, выполненные флаги, текущее время).  
- Загружает при старте или по запросу из меню.  
- Сохранение вызывается в ключевых точках (например, при переходе между сценами, после важных событий).

## 4. Правила добавления новых фич

1. Всегда начинай с создания папки в `Scripts/Systems/` (если это новая система) или дополняй существующую.  
2. Определи нужные ScriptableObject-классы в `Scripts/Data/`, а их экземпляры создавай в `Assets/Data/`.  
3. Для взаимодействия с другими системами используй `EventBus` — публикуй события и подписывайся на них.  
4. Если новый объект должен быть интерактивным — создай префаб в `Prefabs/Interactables` и прикрепи скрипт `Interactable` с методом `OnInteract()`.  
5. Для любой логики, зависящей от времени, подписывайся на события `TimeManager`.  
6. Для UI-элементов — создавай панель в `Scenes/UI/` (аддитивная сцена) или префаб в `Prefabs/UI/` и управляй через `UIManager`.  
7. Не забывай обновлять `GameManager` и `SaveSystem`, если новая система имеет сохраняемое состояние.

## 6. Технические ограничения

- Все исходные файлы `.blend`, `.psd` и т.п. **запрещено** хранить в этом репозитории. Только экспортированные `.fbx`, `.png`.  
- Модели и текстуры должны быть оптимизированы (полигонаж, размер текстур) под производительность.  
- Для анимаций используй Animator Controller и Animation Clips, храни их в `Art/Animations/`.  
- Катсцены настраивай через Timeline и Cinemachine, не создавай отдельные сцены для них.  
- Все публичные поля, которые будут настраиваться дизайнером, помечай `[SerializeField]` и давай им осмысленные имена.  
- Код документируй XML-комментариями для публичных методов и классов.
